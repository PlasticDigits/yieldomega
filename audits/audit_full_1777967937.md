## Executive Summary

No critical unprivileged path was identified that directly lets an attacker move core protocol funds from TimeCurve, RabbitTreasury, FeeRouter, vesting, or the Kumbaya buy router under the assumptions below. The contract layer has materially addressed many issues from the prior smart-contract audit: ERC20 balance-delta parity on ingress, TimeCurveBuyRouter net refunds, explicit post-end gates, event coverage for indexer observability, 300-day sale cap checks, buy-router sale-live fail-fast, and residual routing for empty or partial podium categories.

The most important remaining risks are cross-layer and operational:

- Referral registration lacks the wallet-session continuity guard already used for TimeCurve buys. A wallet/account switch during the approve/register flow can burn/register from a different account than the UI closure expects and can cache the plaintext referral code under the wrong wallet.
- The indexer can run with a registry/chain mismatch or silently skip invalid addresses. In production that should fail closed, because the frontend and agents may otherwise consume a partial or wrong-chain read model.
- Public API handlers return raw database error strings on 500s, which is unnecessary information disclosure.
- Several referral aggregate endpoints use `LOWER(column)` predicates on address columns, creating avoidable index-defeating queries as referral volume grows.

Governance/admin authority is treated as trusted for this report per maintainer instruction. This review does not classify intended governance power, UUPS upgrade authority, owner rescue functions, or final signoff gates as vulnerabilities by themselves. The report still reviews governance-controlled paths for implementation mistakes, misconfiguration risk, and user-facing disclosure.

## Assumptions and Trust Boundaries

### Trusted Assumptions

- Governance/admin roles are trusted and expected to operate in good faith.
- Production governance keys are assumed to use operational controls appropriate for value movement, even when the code does not enforce multisig/timelock by itself.
- Canonical deployments use ERC1967 proxy addresses, not implementation addresses.
- CL8Y / DOUB / USDm / WETH integrations use expected ERC20 semantics unless explicitly tested as non-standard.
- Kumbaya router/quoter addresses are trusted when configured for the correct chain.
- Frontend is a static client and does not hold private keys or server-side secrets.

### Authority Model

The repo correctly documents and mostly implements the central trust boundary:

- Contracts are authoritative for fund movement, sale state, timers, winners, CHARM weights, claims, fee routing, and treasury accounting.
- The indexer is a derived event read model. It can mislead users if wrong, stale, or compromised, but should not be able to move funds.
- The frontend is a UX and wallet orchestration layer. It must not invent authority from indexer data; writes still land against contracts and wallet chain state.

## Severity Findings

### M-01: Referral registration lacks wallet-session drift protection across approve/register

Affected code:

- `frontend/src/pages/referrals/ReferralRegisterSection.tsx`
- Existing guard pattern: `frontend/src/lib/walletBuySessionGuard.ts`

`ReferralRegisterSection.onRegister` checks the target chain before starting, reads allowance for the React-closure `address`, optionally submits `approve`, waits for the receipt, then submits `registerCode`. Unlike TimeCurve buys, it does not capture and re-check the connected account and chain after each await.

This creates a race if the wallet account or network changes after the initial checks. Possible outcomes:

- An approval can be requested from the newly active wallet while the UI still reasons about the old `address`.
- `registerCode` can register the normalized code for a different wallet than the one the UI closure will later cache with `setStoredMyReferralCodeForWallet(address, normalized)`.
- If the old wallet had allowance but the new wallet does not, the registration can fail in a confusing way after the UI skipped approval.
- If the new wallet approves/registers successfully, local storage may show the code under the wrong wallet until reads invalidate it.

This is not a protocol fund-drain issue: `ReferralRegistry.registerCode` burns from `msg.sender`, the burn amount is exact, and the contract owner mapping follows the actual caller. The risk is user confusion, wrong-wallet referral ownership, and avoidable CL8Y burn/signature prompts during wallet drift.

Recommendation:

- Reuse the `walletBuySessionGuard` pattern for referral registration.
- Capture a session snapshot immediately after validating inputs and before the first async read/write.
- Assert unchanged after allowance read, after approve receipt, before register, after register receipt, and before local storage writes.
- Add a Vitest test around a pure helper or a mocked registration flow, and add a manual QA row mirroring `INV-BUY-SESSION-144`.

Suggested severity: Medium. It can cause wrong-account ownership and burns in a realistic UX race, but does not bypass contract authorization.

### M-02: Indexer production config should fail closed on registry/chain/address mismatch

Affected code:

- `indexer/src/config.rs`
- `indexer/src/ingestion.rs`
- `indexer/.env.example`

`Config::from_env` warns when `ADDRESS_REGISTRY.chain_id` differs from `CHAIN_ID`, but continues. `AddressRegistry::index_addresses` silently skips invalid addresses and ingestion idles forever when the resulting address list is empty.

This is acceptable for local bring-up, but not for production. A wrong registry, wrong chain id, implementation address, invalid address, missing buy router, or stale deploy block can produce a partial or entirely wrong read model. Since the frontend and agents consume this indexer for history, leaderboards, TimeCurve clocks, and referral metrics, a production misconfiguration can materially mislead users even though contracts remain authoritative.

Recommendation:

- When `INDEXER_PRODUCTION=1`, fail startup if:
  - registry `chain_id != CHAIN_ID`;
  - `ADDRESS_REGISTRY_PATH` is missing;
  - required deployed protocol addresses are missing, invalid, zero, or not parseable;
  - the resolved index address set is empty;
  - `deploy_block` is zero for a non-local chain;
  - optional-but-shipping surfaces such as `TimeCurveBuyRouter` are expected by the frontend env but absent from the registry.
- Keep permissive warn-and-idle behavior for development.
- Add unit tests for production registry validation and an operator checklist that compares proxy addresses with deployment artifacts.

Suggested severity: Medium. It cannot move funds, but it can cause public wrong-chain/wrong-contract history and timer displays.

### L-01: Public indexer API returns raw database error strings

Affected code:

- `indexer/src/api.rs`

Many API handlers return:

```text
StatusCode::INTERNAL_SERVER_ERROR, Json(json!({ "error": e.to_string() }))
```

Raw `sqlx` error strings can expose table names, column names, SQL fragments, network details, constraint names, or migration drift. The information is not needed by browser clients and increases the usefulness of the public API as a reconnaissance target.

Recommendation:

- Return a generic public body such as `{ "error": "internal server error" }`.
- Log the detailed error with `tracing::warn!` / `tracing::error!` and include a request/trace id if available.
- Add a small helper, for example `internal_error_response(context, err)`, so the pattern cannot drift endpoint-by-endpoint.
- Add an API unit test that a forced query failure does not echo raw SQL details.

Suggested severity: Low. This is information disclosure and hardening, not an immediate data-integrity break.

### L-02: Some referral API address predicates use `LOWER(column)` and can defeat indexes

Affected code:

- `indexer/src/api.rs`

The WarBow hardening invariant explicitly avoids `LOWER(column)` on indexed address columns and compares lowercase-bound parameters to stored lowercase addresses. Several referral endpoints still use patterns like:

- `WHERE lower(referrer) = lower($3)`
- wallet summary subqueries with `WHERE lower(referrer) = lower($1)` and comparable predicates.

Since persisted addresses are already generated by lowercase `addr_hex`, these predicates can be rewritten to compare the plain column against a lowercase bound parameter. This keeps query plans index-friendly as referral rows grow.

Recommendation:

- Replace referral predicates with `WHERE referrer = $n` / `WHERE buyer = $n` after `bind_addr_lower`.
- Add or confirm btree indexes on `idx_timecurve_referral_applied(referrer)` and `idx_timecurve_referral_applied(buyer)`.
- Add integration tests or query-plan smoke for the hot referral leaderboard/wallet endpoints.

Suggested severity: Low. This is an availability/performance hardening issue.

## Contract-Layer Review

### TimeCurve

Positive controls observed:

- `startSaleAt` is owner-only, single-use, rejects zero/past epochs, checks sale token funding, and clamps the first deadline to the 300-day cap.
- `buy`, `buyFor`, WarBow burns, and flag claims enforce sale live checks, `ended == false`, inclusive deadline semantics, and `MAX_SALE_ELAPSED_SEC`.
- `buyFeeRoutingEnabled` gates buy and WarBow CL8Y burn paths.
- ERC20 ingress uses balance-delta parity before accounting and fee routing.
- Referral self-use is rejected; referral bonuses alter CHARM weight, not reserve transfers.
- Presale bonus is isolated to extra CHARM weight for recognized vesting beneficiaries.
- Per-wallet cooldown limits repeat buys from the same wallet while preserving deterministic same-block ordering for distinct wallets.
- `redeemCharms` sets `charmsRedeemed` before transfer and is gated by `charmRedemptionEnabled`.
- `distributePrizes` requires ended state, non-duplication, payout signoff, and for non-zero podium pool requires `warbowPodiumFinalized`.
- Empty podium pool settlement emits a distinct event and latches `prizesDistributed`.
- Empty/partial category slices are forwarded to `podiumResidualRecipient` instead of being silently stranded.
- `refreshWarbowPodium` reverts after sale end, preventing post-end public unlatching of owner finalization.

Residual risks and accepted game mechanics:

- Last-buy, timer extension, WarBow ranking, flag claims, and end-of-sale actions are intentionally ordering-sensitive. Builders/searchers can influence inclusion order.
- Timer griefing by repeated minimum-valid buys is possible if the economics make it worthwhile. The cost is onchain and paid through fee routing; this is game design, not a hidden offchain rule.
- WarBow podium finalization depends on owner-provided candidate sets after sale end. Governance is trusted, but runbooks should require candidate-set derivation from chain/indexer evidence and archive the inputs used for finalization.
- Redemption weights can be diluted by referral and presale bonuses by design. UX should keep explaining CHARM weight versus direct CL8Y spend.
- Non-rebasing, standard ERC20 assumptions still matter for reserve assets even though fee-on-transfer ingress is rejected.

### TimeCurveBuyRouter

Positive controls observed:

- Fails before swap when sale is unscheduled, ended, not live yet, or past deadline.
- Rechecks current CHARM bounds and current price before computing exact CL8Y out.
- Validates v3 exact-output path first token as CL8Y and last token as WETH/stable.
- ETH and stable refunds use balance snapshots so pre-seeded WETH/stable cannot be drained by the current caller.
- Stable ingress requires exact balance delta.
- CL8Y dust goes to protocol treasury; owner rescue functions are explicit and evented.
- `Ownable2Step` is used for router ownership lifecycle.

Residual risks:

- Kumbaya path quality and pool pricing are external DEX risks. Users can still be sandwiched up to `amountInMaximum` / frontend slippage.
- `msg.value == maxIn` for ETH single-tx buys can lock failed quote/gas user experience to the quality of frontend quotes and swap deadline freshness, though transaction reverts protect net funds if the swap path fails.

### RabbitTreasury

Positive controls observed:

- Initialization rejects zero reserve, zero DOUB, zero epoch, invalid min efficiency, and zero admin.
- Burrow curve parameters are bounded in initializer and setters per documented envelope.
- Redeemable backing and protocol-owned backing are separated.
- Deposits and fees use balance-delta parity.
- Withdrawals burn DOUB via holder allowance rather than a third-party burner role.
- `Pausable` is available for deposit/withdraw paths where implemented.
- Epoch finalization emits canonical health events for indexer and frontend.

Residual risks:

- Repricing remains complex and should keep invariant fuzz plus simulations aligned.
- Rebasing reserve assets can still desynchronize internal accounting if balances change without a protocol function call. The repo documents this as discouraged.
- Public messaging should avoid “bank deposit” framing; reserve/DOUB mechanics are game accounting with rule-bound withdrawals.

### FeeRouter and Sinks

Positive controls observed:

- `FeeRouter.initialize` rejects zero admin and zero sink destinations.
- Sink weights must sum to 10,000 bps.
- `distributeFees` is permissionless but only for allowlisted distributable tokens; wrong assets require governed rescue.
- Rounding remainder goes to the last sink.
- Updates emit old and new sink arrays.

Residual risks:

- A reverting sink or blocked token transfer can halt fee distribution for that token. Under trusted governance this is recoverable by sink/token configuration, but should be in runbooks.
- Anyone can call `distributeFees` when the router holds allowlisted token balance; callers must pass the measured ingress amount. This is documented and compatible with TimeCurve's measured `received`.

### ReferralRegistry

Positive controls observed:

- Code normalization is deterministic: lowercase alphanumeric, length 3-16.
- Code uniqueness and one-code-per-owner checks occur before CL8Y burn.
- Burn uses balance-delta parity at the burn address.
- Front-running of plaintext `registerCode` is explicitly documented and disclosed.

Residual risks:

- Registration ordering is mempool-visible and first-included wins. This is accepted, but users should continue seeing the disclosure before burning CL8Y.
- The frontend session-drift issue is the main actionable weakness, not the contract.

### DoubPresaleVesting

Positive controls observed:

- Unique non-zero beneficiaries and non-zero allocations.
- Sum of allocations must equal required total.
- Vesting starts once and only when funded.
- Claims require both vesting started and `claimsEnabled`.
- Claimed accounting updates before transfer.
- Vesting-token rescue is limited to excess over outstanding claims; non-vesting token rescue is separate and evented.

Residual risks:

- `reserveDoubForOutstandingClaimsWad` and rescue are O(n) over beneficiaries. This is acceptable for a fixed presale set, but very large beneficiary arrays would create operational gas concerns.

### NFT Metadata

Positive controls observed:

- Onchain traits remain the gameplay authority.
- Docs disclose mutable offchain `baseURI` / `tokenURI` trust.

Residual risks:

- Offchain metadata can change if admin updates `baseURI`. This is accepted under trusted governance but should not be marketed as immutable without an onchain freeze mechanism.

## Indexer Review

Positive controls observed:

- `ingestion::run` processes each block in one SQL transaction: decoded event rows, `indexed_blocks`, and `chain_pointer` commit together.
- Reorg detection compares parent hash to the stored pointer and rolls back to a common ancestor.
- `rollback_after` deletes every documented `idx_*` table in a single transaction and then updates the pointer.
- `MAX_REORG_DEPTH` produces a fail-stop condition for deep reorgs instead of silently inventing state.
- API pagination clamps limits to 200 and offsets are bounded at zero.
- `INDEXER_PRODUCTION=1` requires configured CORS origins and rejects placeholder database URLs.
- The decoder/persist layer includes current TimeCurve, buy-router rescue, FeeRouter rescue, RabbitTreasury, ReferralRegistry, PodiumPool, vesting, and NFT event families.

Threats and residual risks:

- RPC trust is central: the indexer accepts the configured RPC's block hashes, timestamps, and logs.
- Chain timer is a convenience layer. A stale or malicious indexer can make the UI display the wrong phase or countdown, but contract writes still enforce `saleStart`, `deadline`, and `ended`.
- `BuyViaKumbaya` metadata is optional and only ingested when `TimeCurveBuyRouter` is in `ADDRESS_REGISTRY`. Missing registry wiring hides entry-pay attribution while canonical `Buy` rows remain.
- Last-buy prediction before sale end is derived from indexed buys and can be stale around reorgs or indexer lag. The onchain finalized podium remains authoritative after `endSale`.
- Public endpoints with expensive aggregate queries should be monitored as data grows.

## Frontend Review

Positive controls observed:

- `chainMismatchWriteMessage` preflights writes against the build target chain.
- TimeCurve buy flows capture wallet session after fresh submit-time sizing and assert no account/chain drift across awaits.
- Kumbaya single-tx buys also receive and enforce the session snapshot.
- Default ERC20 approvals are exact or slippage-bounded; unlimited CL8Y approval is opt-in.
- Kumbaya routing fails closed for missing chain/router/quoter/WETH/USDM and for env/onchain buy-router mismatch.
- Swap deadlines come from chain time, not `Date.now`.
- TimeCurve phase uses `ledgerSecIntForPhase`, preferring hero timer chain time when available, and respects the inclusive deadline second.
- Indexer API helpers catch malformed JSON and return `null`.
- Wrong-chain barriers exist on TimeCurve, referrals, vesting, Arena, and WarBow write surfaces.
- `wagmi-config.ts` declares one target chain only, preventing incidental mainnet/default transport probes in local QA.

Threats and residual risks:

- Frontend env is public and build-time. A wrong `VITE_*` address can mislead users or point writes at the wrong deployment for the target chain. Write gates protect chain id, not address correctness.
- `VITE_E2E_MOCK_WALLET=1` must never be present in production builds. The env example documents this, but the current `check-frontend-vite-env.sh` validates required addresses and chain settings, not “forbidden in prod” values.

- Referral storage is local/session UX only and must not be treated as ownership proof.
- Outbound explorer/DEX links are public URL trust surfaces; `rel="noreferrer noopener"` is used in reviewed address-link components by documentation.

## TimeCurve Manipulation Review

### Ordering and MEV

The TimeCurve sale is intentionally sensitive to transaction ordering:

- Last-buy podium depends on inclusion order.
- Same-block distinct-wallet buys are ordered by transaction/log ordering.
- Flag claim windows can be interrupted by another buy after the silence window.
- Defended streak and clutch bonuses are boundary-sensitive.
- WarBow steals/revenge/guard can change the Battle Points podium shortly before sale end.

This is not fully preventable without changing the game. Current mitigations are deterministic onchain rules, per-wallet cooldown, explicit burn costs for WarBow actions, and public events. Recommended posture: document proposer/builder ordering as part of the game, avoid UI promises of “guaranteed” winner status before finalization, and show block/tx context on high-stakes displays.

### Timer Griefing and Sale Extension

Small valid buys can extend or hard-reset the timer within configured caps. The controls are:

- live sale checks;
- min/max CHARM bounds;
- per-wallet cooldown;
- timer cap;
- 300-day wall-clock cap;
- fee routing of every buy.

Residual risk: a well-funded actor can pay to extend the sale or manipulate visible urgency. That is an economic/game-design risk rather than a hidden implementation bug.

### Scheduled Start and Pre-Open Manipulation

Contracts reject buys and WarBow interactions before `saleStart`; the buy router also fails before swapping. Frontend phase and pre-open countdown use indexer/RPC-derived chain time. A stale indexer can show the wrong countdown, but writes fail onchain until live.

Recommended posture: keep the hero timer explicitly labeled as display and use contract errors as final truth.

### Inclusive Deadline and End Sale

Contracts allow buy/WarBow mutations when `block.timestamp <= deadline` and require `block.timestamp > deadline` for `endSale`. Frontend `derivePhase` mirrors this with `ledgerSecInt > deadlineSec`. This is correct and avoids closing the UI one second too early.

### WarBow Podium Manipulation

WarBow BP can both increase and decrease. The repo added refresh/finalize controls to address snapshot drift. Remaining manipulation potential is mostly strategic:

- players can buy/steal/guard/revenge near deadline;
- owner finalization must include a superset of BP candidates;
- indexer lag can hide recent BP changes from the UX.

Recommended posture: finalization runbooks should archive candidate derivation, input arrays, and resulting podium reads before prize distribution.

### Referral and Presale Weight Dilution

Referral and presale bonuses mint extra CHARM weight without equivalent additional CL8Y spend. This dilutes the eventual DOUB per CHARM for other holders by design. The frontend already prefers CL8Y-at-launch projections and distinguishes CHARM weight from direct balance. Keep that disclosure.

### Kumbaya Route and Quote Manipulation

Kumbaya exact-output paths cap input with slippage and deadline, but a user can still pay up to `maxIn` in adverse pool conditions. The contract validates path endpoints, not price quality. Price quality is frontend/quoter-dependent and subject to normal DEX MEV.

Recommended posture: retain quote loading gates, keep slippage visible, avoid stale quote CTAs, and consider warning copy for low-liquidity routes.

## Common Attack Class Coverage

| Attack class | Current posture |
|---|---|
| Reentrancy | Core value-moving paths sampled use `nonReentrant` or checks-effects-interactions. |
| Fee-on-transfer ERC20s | Major ingress paths use balance-delta parity and revert on mismatch. |
| Rebasing ERC20s | Still discouraged for reserve assets; balance drift can occur outside protocol calls. |
| Front-running / MEV | Acknowledged as part of TimeCurve/referral mechanics; deterministic onchain rules but no prevention. |
| Timestamp manipulation | Uses `block.timestamp` with documented inclusive boundaries and 300-day cap; miner/proposer timestamp latitude remains chain-level. |
| Wrong-chain writes | Frontend preflight and barriers present for major write paths. |
| Wallet account drift | Strong for TimeCurve buys; missing for referral registration. |
| Indexer reorgs | Transactional ingest and rollback coverage present; deep reorg fail-stop at 128 blocks. |
| API abuse / info leak | Pagination clamped, but raw SQL errors and some index-defeating predicates remain. |
| Supply chain | Static frontend, lockfiles, CI, Gitleaks, Slither workflow; production env validation can improve. |
| User deception | Many UX disclosures exist; continued emphasis needed for derived leaderboards/timers and referral ordering. |

## Recommended Remediation Queue

1. Add wallet-session drift guards to `ReferralRegisterSection.onRegister`.
2. Make production indexer registry validation fail closed for chain/address/deploy-block errors.
3. Replace raw API 500 error bodies with generic messages and server logs.
4. Rewrite referral address predicates to compare lowercase-bound values against stored lowercase columns.
5. Add frontend production env validation that rejects `VITE_E2E_MOCK_WALLET=1`.
6. Add release evidence templates that record proxy addresses, chain id, deploy block, indexer schema, frontend build id, and command outputs.
7. Run and archive the full matrix before any mainnet launch, including optional Anvil E2E and fork smoke where applicable.

## Conclusion

The onchain layer is relatively well-hardened for the known TimeCurve/Rabbit/FeeRouter threat model, assuming trusted governance and standard production operations. The highest-value next work is cross-layer hardening: make frontend referral writes as session-safe as TimeCurve buys, make production indexer configuration fail closed, reduce API information leakage, and keep performance-sensitive public endpoints index-friendly.

TimeCurve remains inherently adversarial around ordering, timers, podiums, and WarBow. The code handles the main boundary conditions reviewed here; the remaining risk is mostly economic/game-strategy behavior plus the accuracy and clarity of derived UI/indexer displays.
